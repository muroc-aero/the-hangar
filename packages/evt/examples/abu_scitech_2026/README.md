# ABU SciTech 2026 -- non-ABU case-study reproduction

Reproduces the non-ABU result tables from:

> "Autonomous Battery Units as an Enabling Technology for Urban Air
> Mobility," AIAA SciTech 2026.

The paper evaluates three eVTOL aircraft over a fully crossed matrix:

- **3 vehicles**: Archer Midnight, Joby S4, Supernal S-A2
- **2 cruise altitudes**: 1500 ft, 3000 ft
- **3 mission ranges**: 30, 45, 60 miles
- = **18 cases**

The 18 `cfg/*.json` files are the upstream case configs at the pinned evtolpy
ref, vendored here (the same convention `mission_segments/cfg/test-all.json`
uses) so the example is self-contained and all three lanes read identical
inputs.

## Scope

This reproduces the **non-ABU** quantities the paper's case study is built on:
sized MTOW, mission energy, and peak electric power. The paper's headline
**ABU** analysis (autonomous-battery-unit economics, charging, queuing) is
**out of scope for evt v1** (see `packages/evt/CLAUDE.md`).

## Lanes

The same three-lane pattern as `mission_segments`, extended with a declarative
study lane:

- **`lane_a/`** -- the paper's own method: the direct evtolpy API. `case_study.py`
  builds an `Aircraft` from each vendored config and harvests the headline
  metrics. This is the ground truth.
- **`lane_b/`** -- the hangar-evt tool layer. `run_all.py` applies the five
  config sections through the section setters and runs `run_mission_analysis` /
  `run_sizing` in-process; `case_study.json` is the equivalent `evt-cli
  run-script` workflow for one representative case.
- **`lane_c/`** -- the declarative SDK **study layer** driving the registered
  `evt` runner. `range_study.yaml` (generated by `gen_study.py`) sweeps the
  range axis (the one scalar axis -- range is encoded purely in
  `mission.cruise_s`) for Archer Midnight at 1500 ft. The altitude and vehicle
  axes touch many coupled keys, so they are data-driven axes carried by Lanes
  A/B over the 18 configs rather than scalar sweeps.

All three lanes agree to floating-point round-off; any drift is a wrapper bug.
`figures/reproduced/lane_parity.png` (from `pipeline/compare_lanes_plot.py`)
draws this directly: Lane A as a line, Lane B (the cli/mcp tool path) as `x`
markers on top of it, with the max |delta| over all 18 cases in the title
(0.00e+00 -- exact). The two Joby S4 60-mile points are absent in *both* the
line and the markers, so the divergence below is visibly upstream, not a
wrapper artifact.

## Status / fidelity vs the paper

`pipeline/compare_to_paper.py` compares the reproduced grid (Lane A at the
pinned evtolpy ref) against values transcribed from the upstream summary
workbook (`result-summary/weight-analysis/mtow_iteration_1500_3000_ft.xlsx`).

- **Mission energy: reproduced exactly.** All 18 cases match the paper's
  published "Total Mission Energy (without resizing)" to its 2-decimal
  precision (delta <= 0.01%). The energy model is unchanged at the pinned ref.
- **Sized MTOW: a documented upstream drift.** evtolpy's MTOW resizing loop has
  changed since the paper, so the reproduced final MTOW runs ~1.2% high for
  Archer Midnight and Supernal and up to ~12.6% high for Joby S4. The two Joby
  S4 60-mile cases that converged in the paper now hit evtolpy's divergence
  safeguard at the pinned ref and are reported as `diverged` rather than forced
  to a number.

This split is the point of pinning + golden tests + provenance: the energy
physics is stable across the upstream bump, the resizing loop is not, and the
demo surfaces exactly where.

## Running

```bash
# One-command full reproduction: grid CSV + figures + fidelity table.
bash packages/evt/examples/abu_scitech_2026/run_case_study.sh

# Or the steps individually:
uv run python packages/evt/examples/abu_scitech_2026/pipeline/aggregate.py
uv run python packages/evt/examples/abu_scitech_2026/pipeline/plotting.py
uv run python packages/evt/examples/abu_scitech_2026/pipeline/compare_lanes_plot.py
uv run python packages/evt/examples/abu_scitech_2026/pipeline/compare_to_paper.py

# Lane C on its own (declarative study via the evt runner):
uv run python packages/evt/examples/abu_scitech_2026/lane_c/compare_to_lane_a.py

# Lane B one representative case through the CLI:
evt-cli --pretty run-script packages/evt/examples/abu_scitech_2026/lane_b/case_study.json
```

## Layout

```
packages/evt/examples/abu_scitech_2026/
  README.md                 <- this file
  run_case_study.sh         -- grid CSV + figures + fidelity table
  shared.py                 -- case enumeration, golden + paper reference values

  cfg/                      -- the 18 vendored upstream case configs

  lane_a/case_study.py      -- ground truth: direct evtolpy API
  lane_b/run_all.py         -- evt tools, in-process, all 18 cases
  lane_b/case_study.json    -- evt-cli run-script for one representative case
  lane_c/gen_study.py       -- emits the declarative study spec from cfg/
  lane_c/range_study.yaml   -- generated: the evt-runner study (range axis)
  lane_c/compare_to_lane_a.py -- run the study, check vs Lane A

  pipeline/aggregate.py     -- 18 cases -> results/case_study_grid.csv
  pipeline/plotting.py      -- figures/reproduced/{sized_mtow,mission_energy}.png
  pipeline/compare_lanes_plot.py -- figures/reproduced/lane_parity.png (A vs cli/mcp)
  pipeline/compare_to_paper.py -- fidelity table vs the paper xlsx

  results/case_study_grid.csv  -- the reproduced 18-row table
  figures/reproduced/          -- sized_mtow.png, mission_energy.png, lane_parity.png

  tests/                    -- parity (A/B/C) + golden (pinned at the ref)
```

## Tests

```bash
# Fast subset (parity A/B/C + golden subset); not in the default suite.
uv run pytest packages/evt/examples/abu_scitech_2026/tests/ -m "not slow"

# Full 18-case golden grid.
uv run pytest packages/evt/examples/abu_scitech_2026/tests/
```

The golden tests pin Lane A at the current evtolpy ref, so a pin bump that
shifts the physics (energy or resizing) is caught. The `PAPER` values in
`shared.py` are the published targets the fidelity table compares against.
